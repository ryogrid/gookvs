## Codebase Patterns
- TiKV workspace has 80+ crates organized under cmd/, components/, src/, tests/, fuzz/
- Key dependency layers: foundational (codec, keys) → utils (tikv_util) → engine abstraction (engine_traits) → consensus (raftstore, batch-system) → txn (storage/txn) → server
- Thread pools: gRPC (grpcio), read pool (YATP), raft batch system (custom FSM), scheduler worker pool, background workers (MPSC-based)
- Request flow: gRPC → Storage → TxnScheduler (latches) → RaftKv → Raft replication → Apply → RocksDB
- PD client uses bidirectional gRPC streams for heartbeats and TSO
- Configuration hierarchy mirrors component structure; online_config crate supports runtime changes
- Raftstore uses two FSM types: StoreFsm (one per store) and PeerFsm (one per region), driven by batch-system Router
- Raftstore-v2 reorganizes code into operation/ modules (command/admin/split.rs, ready/snapshot.rs, life.rs)
- Transaction actions (prewrite, commit, rollback, check_txn_status, gc) are in src/storage/txn/actions/ as standalone functions
- MvccTxn (txn.rs) accumulates writes; MvccReader (reader/) handles reads — write and read paths are separate abstractions
- ConcurrencyManager in components/concurrency_manager/ tracks max_ts and in-memory locks for async commit correctness
- Latch system in src/storage/txn/latch.rs uses sorted key hashes for deadlock-free serialization
- GC has two modes: GcManager-driven per-region GC and RocksDB compaction filter GC (compaction_filter.rs)
- Resolved TS computed per-region in components/resolved_ts/src/resolver.rs using BTreeMap-based lock tracking
- Coprocessor batch execution uses 6 tidb_query_* crates; expressions are RPN (flat postfix, not trees) evaluated with stack-based batch evaluator
- Coprocessor V2 plugins are cdylibs loaded via libloading with hot-reload filesystem watcher; interact through RawStorage trait only
- gRPC dispatch uses 3 macros: handle_request! (reads), txn_command_future! (txn writes), handle_cmd! (batch_commands sub-dispatch)
- Flow control is multi-layered: read pool busy threshold (EWMA), scheduler memory quota (256MB), write flow controller (discard ratio + rate limiter), Raft message rejection on memory pressure
- Status server is a separate Hyper HTTP server on status_addr (default :10080), not part of the gRPC server
- RaftClient uses lock-free ArrayQueue per connection with seahash(region_id) for connection selection; batch_raft is default with raft as fallback
- PD client has V1 (distributed reconnection) and V2 (centralized reconnection loop) implementations; TSO uses bidirectional streaming with batched requests
- CDC, backup-stream, and resolved_ts all register as observers on the same CoprocessorHost; ObserveLevel (LockRelated vs All) determines what data is captured
- Resolved TS Resolver uses dual tracking: BTreeMap for normal locks, separate HashMap+TxnStatusCache for large transactions
- Backup-stream TwoPhaseResolver wraps resolved_ts Resolver with phase 1 (initial scan) buffering incremental locks in future_locks vec
- Full backup uses direct engine snapshots (NOT raftstore observer pattern); backup-stream uses observer pattern for incremental capture
- ExternalStorage trait (components/external_storage/) provides unified S3/GCS/Azure/local/HDFS abstraction shared by backup and backup-stream
- Resource control uses virtual-time priority scheduling in YATP read pool; token bucket only for background tasks
- Encryption: two-tier key hierarchy (master key → key dictionary → data keys), file_dict is plaintext, key_dict encrypted
- OnlineConfig derive macro with skip/hidden/submodule attributes controls runtime changeability of config fields
- ConfigController dispatches changes per-module via ConfigManager trait; validates before dispatch, persists after
- Log routing by slog tag: "slow_log*"/"rocksdb_log*"/"raftdb_log*" routed to separate drains via LogDispatcher
